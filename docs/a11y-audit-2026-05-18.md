## Root causes (small number of patterns, huge leverage)

### 1. Sidebar nav group headers lack accessible names ‚Äî **affects every page** (7 violations √ó 23 pages = 161 reported instances)

Each `<div class="menu-group-header" role="button" data-tooltip="‚Ä¶">` had no `aria-label`, so screen readers announced "button" with no context. The 7 groups are Assets, Finance, Materials & Purchasing, Work Management, Projects, AI & Integrations, Administration.

**Fix in this PR:** added `aria-label="{group name} navigation group"` to every header in `_ModernLayout.cshtml`. **One file, eliminates 7 violations from every page in the app.**

### 2. Filter toolbar `<select>` elements without accessible names ‚Äî list pages

`<select name="statusFilter">` and friends had no `<label>` and no `aria-label`. Screen readers would announce "combobox" with no indication of what's being filtered.

**Fix in this PR:** added `aria-label` to all three filter selects on `/Materials/Items` (statusFilter, typeFilter, vendorId). Same pattern queued for other list pages in follow-ups.

### 3. Form inputs use placeholder instead of `<label>` ‚Äî edit pages

`<input placeholder="Enter part number">` is invisible to screen readers (and disappears when the user starts typing). WCAG requires a persistent `<label for="‚Ä¶">` or `aria-label`.

**Status:** documented; remediation queued as PR #116d.23a. Touches ItemEdit, VendorEdit, every other edit/create surface. Scoped separately because it requires careful UX review per field.

### 4. Color contrast ‚Äî single tab element

`#tab-basics` text-on-background ratio falls below the 4.5:1 WCAG AA threshold.

**Status:** documented; one-line CSS fix queued as PR #116d.23b.

## What PR #116d.23 ships
