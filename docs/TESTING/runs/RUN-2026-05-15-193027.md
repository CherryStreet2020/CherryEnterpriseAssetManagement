**Where it lives:** AP invoice approval JE generation — likely `Services/AccountsPayable*` and/or `Pages/AccountsPayable/Details.cshtml.cs` (`OnPostApproveAsync`).

**Repro (exact steps I ran):**

5. `/AccountsPayable` → "New Invoice". Fill vendor=3M INDUSTRIAL, invoiceNumber=E2E-INV-193027, invoiceDate=today, amount=$50 → "Create Invoice". Invoice id 7 in PENDINGAPPROVAL.
6. On the invoice detail there are two action buttons: "Auto-Match to PO" and "Approve". **I clicked "Approve" without first clicking "Auto-Match to PO".** Status → APPROVED. No warning. No required match.
7. Open `/Journals`. The approval JE `AP-APR-E2E-INV-193027` shows **1 line, $0.00 D / $0.00 C**. Header says "BALANCED" (correctly — zero equals zero), but the JE is empty.
8. The detail page (`/Journals/Details/1794`) confirms: single line, account 2000 "AP — Vendor 69", $0/$0.
9. "Record Payment" on the same invoice → $50 / Check → payment JE `AP-PMT-E2E-INV-193027` posts correctly (2 lines: DR 2000 $50 / CR 1110 Cash $50).

**Net GL effect of the chain:**
| Account | Debit | Credit | Source |
|---|---|---|---|
| 6000 Operating Expenses | $50 | | GR line 1 |
| 2150 GR-IR Clearing | | $50 | GR line 2 |
| 2000 AP — Vendor 69 | $50 | | Payment line 1 |
| 1110 Cash | | $50 | Payment line 2 |

GR-IR Clearing was never relieved (should have been the DR side of approval); AP for Vendor 69 has a **$50 debit balance** for a paid invoice that was never accrued. The accounting is broken. With volume, this corrupts the trial balance.

### DEF-N02 — Asset disposal JE is unbalanced when DisposalExpense > 0 *(P0)*

**Where it lives:** Disposal JE generation — `Services/Assets/AssetDisposalService.cs` (or wherever the `Disposal` JE source is built). The handler is at `Pages/Assets/Dispose.cshtml.cs` (`OnPostAsync`).

**Repro:**
1. Create a new asset: AssetNumber=E2E-AST-193027, Cost=$10,000, InService=today, Life=60mo, SL, etc. → `/Assets/Asset/1075`.
2. Navigate to `/Assets/Dispose/1075`.
3. Fill Proceeds=$3,000, DisposalExpense=$100, leave CreateJournalEntry checked → "Record Disposal".
4. Open `/Journals` → row shows JE `DISP-20260515-E2E-AST-193027`, **5 lines, $10,200.00 D / $10,000.00 C** — off by $200.
5. Detail at `/Journals/Details/1796` confirms — header explicitly says "UNBALANCED":

| # | Account | Description | DR | CR |
|---|---|---|---|---|
| 1 | 1510 | Remove accumulated depreciation - E2E-AST-193027 | $0.00 | $0.00 |
| 2 | 1110 | Cash/AR from disposal (gross) - E2E-AST-193027 | $3,000.00 | $0.00 |
| 3 | 6500 | Disposal expense - E2E-AST-193027 | $100.00 | $0.00 |
| 4 | 1500 | Remove asset cost - E2E-AST-193027 | $0.00 | $10,000.00 |
| 5 | 6510 | Loss on disposal - E2E-AST-193027 | $7,100.00 | $0.00 |

**Math the code did:** Loss = (Cost − AccDep) − (Proceeds − Expense) = 10,000 − 0 − (3,000 − 100) = $7,100. That value is plugged into line 5 as a debit. But the disposal expense (line 3) gets debited *separately* with no offsetting credit anywhere — the cash that was spent for the expense should be credited but isn't.

**Two valid balancing approaches:**
- **A:** Net it. Make line 2 "Net cash from disposal" = $2,900 DR (proceeds − expense), drop line 3. Total D = $10,000 = C. ✅
- **B:** Keep gross. Line 2 = $3,000 DR Cash, line 3 = $100 DR Disposal Expense, add line 6 = $100 CR Cash. Total D = $10,200 = C ($10,000 + $100 net cash). ✅

**System behaviour:** The persistence layer accepted the unbalanced JE rather than rejecting it. The "UNBALANCED" badge in the JE header indicates the validation logic *recognizes* the problem but the write happens anyway. There should be an `if (!je.IsBalanced) throw` guard at the persistence boundary.

**Why this wasn't caught yesterday:** The marquee disposal example (`DISP-20260503-AST-00009`) had DisposalExpense=$0 or wasn't computed with the same path. Any historical disposal with non-zero DisposalExpense in your DB is potentially also imbalanced — worth a SQL audit:

```sql
SELECT je."Id", je."EntryNumber",
       SUM(jel."Debit"), SUM(jel."Credit"),
       SUM(jel."Debit") - SUM(jel."Credit") AS variance
FROM "JournalEntries" je
JOIN "JournalEntryLines" jel ON jel."JournalEntryId" = je."Id"
WHERE je."Source" = 'DISPOSAL'
GROUP BY je."Id", je."EntryNumber"
HAVING SUM(jel."Debit") <> SUM(jel."Credit");
```

### DEF-N05 — Work Order created with today's scheduled date displays "OVERDUE" *(P1)*

**Repro:** Same WO as above. ScheduledDate is `2026-05-15`. Detail page header reads `OVERDUE`, not `SCHEDULED` or `OPEN`.

**Likely cause:** The "overdue" predicate probably uses `scheduledDate < DateTime.UtcNow` while comparing only the date portion of `scheduledDate` (00:00 UTC) against a `DateTime.UtcNow` later in the day — so any WO scheduled today is "overdue" by the afternoon. Either compare `scheduledDate.Date < DateTime.UtcNow.Date` (strict less-than on date-only) or treat the schedule as "due by end of scheduled day".

**Suggested fix area:** `Services/Maintenance/*Overdue*` or wherever the WO `IsOverdue` property is computed. Also check the cockpit's overdue count.
